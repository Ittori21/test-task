1) нужно обеспечить высокую доступность сервиса, как бы ее реализовал?
Чтобы обеспечить высокую доступность сервиса, необходимо исключить единую точку отказа и добавить избыточность на всех уровнях:
   1. Развернуть контейнеры с Nginx не на одном, а минимум на 2-3 независимых физических серверах или виртуалках.
   2. Перед веб-серверами установить балансировщик нагрузки (например, HAProxy или еще один Nginx в режиме reverse proxy). Балансировщик будет принимать запросы от пользователей и распределять их между доступными бэкенд-серверами.
   3. Чтобы сам балансировщик не стал единой точкой отказа, их также должно быть два (Master и Backup). Между ними настраивается плавающий (VIP) IP-адрес с помощью утилиты `Keepalived` (протокол VRRP). Если основной балансировщик падает, IP-адрес мгновенно переключается на резервный.
   4. Для автоматического контроля за состоянием контейнеров (Health checks), их перезапуска при падении и удобного обновления без даунтайма логично использовать оркестраторы, такие как Kubernetes

2) тебе надо создать и настроить 5 таких серверов, как бы это реализовал? 
Для этой задачи я бы использовал подход IaC и автоматизацию:
   1. Для автоматического создания пяти виртуальных машин (например, в облаке) я бы использовал Terraform. Он позволяет описать нужные ресурсы в виде кода и развернуть их одной командой.
   2. Для настройки самих операционных систем я бы написал playbook на Ansible. Ansible без установки агентов (через настроенные SSH-ключи) подключится параллельно ко всем 5 серверам и выполнит нужные шаги:
       Установит Docker и UFW.
       Настроит правила файрволла.
       Скопирует нужные файлы конфигурации (nginx.conf), сертификаты и скрипт для бэкапа.
       Добавит задачу бэкапа в cron.
       Соберет и запустит Docker-контейнеры.
   3. Весь процесс сборки образа и триггер запуска Ansible-плейбуков можно автоматизировать с помощью инструментов CI/CD (например, GitHub Actions или GitLab CI), чтобы изменения выкатывались на все серверы автоматически при пуше кода в репозиторий.
